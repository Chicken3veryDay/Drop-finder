### Browser catalog fallback

`BrowserGenerationCache` still uses the constant cache name `dropfinder-client-generation-v1` and the origin-root key `/__dropfinder__/last-complete.json`. `CatalogGenerationClient.initialize()` and `refresh()` accept any structurally complete value returned by that cache without comparing its `manifestUrl` or `publicationBaseUrl` with the current client's expected deployment. The default platform registrar still creates the catalog client without a deployment namespace.

A deterministic key calculation for the documented raw.githack `main` and `gh-pages` paths yields the same key for both:

```text
https://raw.githack.com/__dropfinder__/last-complete.json
```

A generation cached by one path can therefore be activated by the other after a network or publication failure, and detail URLs continue from the foreign cached `publicationBaseUrl`.

### Service-worker generation ownership

The same deployment-identity defect also remains in `cloud_pages/sw.js`:

- `META_CACHE` is the constant `dropfinder-generation-meta-v1`.
- Metadata keys are relative `./prepared.json` and `./active.json` within that origin-wide cache.
- Generation cache names contain only the generation ID: `dropfinder-data-<generation>`.
- `cleanupGenerations()` enumerates every origin cache and deletes every `dropfinder-data-*` cache outside the current worker's keep set.

Separate path-scoped workers on one origin can therefore overwrite active/prepared metadata and delete each other's current or rollback generation caches. Service-worker scope does not namespace Cache Storage.

## Root cause

Persistent browser and service-worker catalog state has generation identity but no stable deployment identity. Origin-only cache ownership is insufficient for branch, preview, CDN-path, or multi-app hosting.

## Recommended smallest complete correction

1. Derive one stable deployment namespace from the canonical manifest URL or publication base.
2. Store and validate that namespace in `BrowserGenerationCache` envelopes before fallback activation.
3. Include the same namespace in service-worker metadata-cache and generation-cache names.
4. Restrict cleanup to the current namespace.
5. Ignore legacy unnamespaced records unless ownership can be proven; do not let one path migrate or delete another path's entries.
6. Add a shared-origin/two-path regression covering browser fallback, service-worker active/prepared metadata, generation cleanup, rollback retention, and detail URL resolution.

A narrow local helper is preferable to a dependency. URL normalization, Cache Storage keys, and hashing primitives already exist in the platform; an external package would add maintenance without solving repository-specific ownership semantics.
